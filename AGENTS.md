# AI 协作开发规范

本文档给参与《姻缘簿：月老牵线局》黑客松开发的 AI 代理阅读。目标是让多个 AI 在同一仓库内并行工作时，能保持产品方向、数据协议、代码边界和交付标准一致。

## 1. 项目基准

本项目已确定采用“月老牵线局”方案。任何开发判断优先参考：

1. `月老牵线_PRD_研发版.md`
2. `月老牵线_PRD_产品版.md`
3. 本文档
4. 具体任务说明

如果具体任务与 PRD 冲突，先指出冲突并等待确认，不要自行改方向。

第一阶段只交付一个可体验闭环：

```text
接案台
  -> 林黛玉委托详情
  -> 候选池选择
  -> 林黛玉 × 孙悟空三回合相亲
  -> 姻缘簿结果记录
```

不要扩展账号、存档、多人社交、复杂养成、全角色完整路线或大量素材系统。

## 2. 开工前必读

每个 AI 代理开始工作前，必须先读取并遵守以下文件：

1. `AGENTS.md`
2. `Agent.md`
3. `月老牵线_PRD_研发版.md`
4. `月老牵线_PRD_产品版.md`
5. 与当前任务直接相关的 README、计划、数据或代码文件

读取后先输出简短开工说明，至少包含：

- 本次任务目标。
- 计划触碰的文件范围。
- 当前所在分支或工作区状态。
- 验证方式。

如果没有先读这些文件，不要开始改代码或文档。

## 3. Git 与分支协作

多人 AI 协作默认使用 Git 隔离工作。若当前目录还不是 Git 仓库，先说明“当前目录未初始化 Git”，不要假装已经提交或切分支。

### 3.1 每人一个分支

每个 AI 或每个独立任务使用独立分支：

```text
agent/<名字或编号>/<任务短名>
```

示例：

```text
agent/a-local-ui/case-desk
agent/b-rules/ending-engine
agent/c-ai-api/prompt-flow
```

不要多人直接在同一个功能分支上改同一批文件。确实需要接手他人分支时，先阅读该分支 changelog 和最近提交。

### 3.2 开工前检查

开始修改前必须执行或等价确认：

```bash
git status --short
git branch --show-current
```

如果发现未提交改动：

- 先判断是不是自己上一轮留下的。
- 不是自己的改动，不要覆盖、删除或格式化。
- 与当前任务冲突时，停下来说明冲突文件和风险。

### 3.3 及时提交

完成一个可验证的小任务后要及时提交。不要把多个无关任务堆成一个大提交。

推荐提交粒度：

- 新增一个页面或组件。
- 新增一批稳定数据。
- 完成一段规则逻辑并通过验证。
- 修复一个明确问题。
- 更新一份文档。

提交前必须运行匹配的验证，并在提交信息中说明真实改动。提交信息建议：

```text
docs: update AI collaboration rules
feat(local): add case desk flow
feat(rules): add ending evaluation
fix(api): add generation fallback
chore(data): add Daiyu Wukong options
```

禁止提交：

- 未验证的代码。
- 密钥或本地私密配置。
- 构建缓存、临时文件、无关格式化。
- 他人未确认的改动。

### 3.4 每个 AI 写自己的 Changelog

每个 AI 必须维护自己的 changelog，避免多人同时编辑同一个日志文件。

推荐位置：

```text
docs/changelog/<agent-name>.md
```

示例：

```text
docs/changelog/agent-a-local-ui.md
docs/changelog/agent-b-rules.md
docs/changelog/agent-c-ai-api.md
```

每次完成一个小任务或提交前，追加一条记录：

```text
时间：2026-06-06 16:30

任务：接案台页面骨架
分支：agent/a-local-ui/case-desk
修改：
- 新增接案台卡片结构
- 标记林黛玉为可接案，其他委托为占位
验证：
- 本地打开 index.html，接案台可见
风险：
- 移动端细节未最终验收
```

Changelog 是给其他 AI 快速接手用的，不要写成宣传文案。

## 4. 双版本边界

项目允许两版并行，但语义必须一致。

| 版本 | 用途 | 约束 | 可做 | 不可做 |
| --- | --- | --- | --- | --- |
| 本地互动空间版 | 黑客松提交与现场稳定演示 | 离线、8MB 内、入口 `index.html` | 预写分支、模板结果、规则数值、生成感 UI | 依赖网络、依赖 API、引入重素材 |
| AI API 完整版 | 展示 AI 核心玩法上限 | 可联网，可独立于互动空间提交 | LLM 生成对话、OS、结尾记录 | 让 LLM 决定核心数值或结局档次 |

两版必须共享同一套角色、选项、数值、记录语义。可以有不同 UI，但不能出现玩法含义不一致。

## 5. AI 协作原则

### 5.1 先同步再动手

开始任何任务前，先明确：

- 当前任务要改哪个交付面：本地版、AI 版、共享数据、文案、样式、文档。
- 成功标准是什么。
- 会触碰哪些文件。
- 需要如何验证。

如果任务描述含糊，先问一个最关键的问题。不要同时抛出一组问题。

### 5.2 小步提交思维

每次改动只解决当前任务：

- 不做未要求的功能。
- 不做无关重构。
- 不顺手整理废案、旧文件或相邻代码。
- 不把单次任务扩成“顺便搭架构”。

每个修改行都应该能追溯到任务目标。

### 5.3 文件所有权

多人 AI 并行时，默认按文件范围隔离：

- 改共享协议、角色数据、数值规则前，必须在交接说明中明确影响范围。
- 同一时间不要让两个 AI 修改同一个核心文件。
- UI、内容、规则、API 接入尽量拆到不同文件或不同模块。
- 如果发现文件已有陌生改动，视为其他成员工作成果，不要回滚。

遇到冲突时，保留现有改动，说明冲突点，让负责人决定。

## 6. 统一数据协议

共享数据语义以研发版 PRD 的 `CaseFile`、`Character`、`Pairing`、`RoundOption`、`Stats`、`RoundRecord`、`SessionRecord`、`DialogueLine`、`InnerThought` 为准。

任何 AI 不得随意改这些字段名和含义。确实需要新增字段时，必须满足：

1. 新字段有明确使用方。
2. 不破坏本地版和 AI 版。
3. 给出默认值或兼容策略。
4. 更新相关文档或类型定义。

核心数值固定为：

```ts
type Stats = {
  romance: number;
  trust: number;
  chemistry: number;
};
```

数值范围默认为 `0-100`。月老表现值为 `kpi`。LLM 只能生成表达内容，不能直接决定 `statDelta`、`kpiDelta`、`endingId`。

## 7. 内容规范

### 5.1 样板线

第一阶段完整样板线固定为：

```text
求助者：林黛玉
候选对象：孙悟空
```

林黛玉不是“求被拯救”，孙悟空不是“无脑搞笑”。所有文案都要保持这条关系张力：

- 林黛玉敏锐、清醒、骄傲，需要被平等接住。
- 孙悟空直来直去、不擅长弦外之音，但愿意认真学。
- 爽点是反差喜剧加治愈成长，不是强行恋爱。

### 5.2 语气

文案可以轻喜剧，但不要恶搞角色人格。避免：

- 把林黛玉写成只会哭、脆弱、需要哄。
- 把孙悟空写成只会闹、完全不可靠。
- 用现代网络梗压过角色本身。
- 强行把所有结局导向爱情成功。

结局允许是爱情、欢喜冤家、知己、礼貌收场或翻车。强扭翻车也是有效体验。

## 8. 规则与结局

核心规则由规则引擎控制。AI 代理实现时应保持可预测：

- 每个选项有明确 `statDelta` 和 `kpiDelta`。
- 每回合记录用户选择、数值变化、角色反馈和摘要。
- 结局按阈值判定，不由文案随机决定。
- KPI 反馈操作质量，不作为唯一胜负条件。

推荐判定优先级：

```text
D 翻车
  -> S 爱情
  -> A 欢喜冤家
  -> B 知己
  -> C 礼貌收场
```

如果需要调阈值，必须说明调参理由，并验证至少一条好结局和一条差结局路径。

## 9. 本地版开发规范

本地互动空间版优先保证稳定和可提交：

- 入口必须是 `index.html`。
- 不依赖外网、远程字体、远程图片、CDN 或 API。
- 总体积控制在 8MB 内。
- 优先使用 HTML、CSS、原生 JavaScript 和轻量本地资源。
- 不引入重型框架，除非负责人明确要求。
- 使用预写分支和模板兜底，避免空结果。

本地版可以通过以下方式模拟 AI 感：

- 打字机效果。
- “月老正在翻姻缘簿”加载态。
- 气泡延迟出现。
- 每回合根据选择拼接不同反馈。

但不要在本地版伪装成真实联网生成。

## 10. AI API 版开发规范

AI API 版用于展示实时生成能力，但必须受规则约束：

- Prompt 输入包含角色卡、当前数值、本回合选择、历史摘要。
- LLM 输出只负责 `dialogue`、`innerThoughts`、`summary`、`finalStory`。
- 输出需要结构化解析或有明确兜底。
- API 失败时给出本地模板结果，不让流程中断。
- 不把密钥写入前端或提交到仓库。

推荐每回合输出限制：

- 2-4 句角色对话。
- 2 条内心 OS。
- 1 句回合摘要。

生成内容必须尊重角色设定和当前数值，不要出现与规则结果相反的文案。

## 11. UI 与交互规范

用户需要在 5 秒内理解自己是月老，在 60-90 秒内完成一局。

界面优先级：

1. 清楚展示当前身份、当前目标和下一步动作。
2. 每次选择后立刻给反馈。
3. 数值变化可见，但不要压过剧情反应。
4. 结果页适合截图分享。

不要做大型落地页、复杂导航、重装饰动画或解释过长的新手引导。第一屏应该直接进入可操作体验。

## 12. 代码规范

### 12.1 通用要求

- 匹配现有技术栈和文件风格。
- 函数和模块只承担一个清楚职责。
- 不为单次使用创建过度抽象。
- 不引入未使用依赖。
- 不提交生成缓存、构建产物、临时文件或密钥。
- 命名要表达业务含义，避免 `data2`、`newLogic`、`tempFinal`。

### 12.2 共享逻辑

规则计算、结局判定、数据定义属于共享核心。修改共享逻辑时必须：

- 给出输入输出说明。
- 覆盖边界值。
- 验证本地版和 AI 版仍可消费。

### 12.3 样式

- 优先清晰、稳定、轻量。
- 适配移动端和桌面端。
- 文本不得溢出按钮、卡片或结果区域。
- 不用大图片堆视觉效果。
- 不用远程字体。

## 13. 测试与验证

每个任务完成前必须执行与改动匹配的验证。

文案或数据改动：

- 检查角色设定是否一致。
- 检查所有引用的 id 是否存在。
- 至少手动跑通林黛玉 × 孙悟空一局。

规则改动：

- 验证 S/A/B/C/D 至少相关路径没有明显误判。
- 验证数值不会低于 0 或高于 100，除非规则明确允许。

UI 改动：

- 验证桌面和移动端关键流程。
- 检查按钮、卡片、弹窗、结果页没有文本重叠。
- 检查离线资源是否可加载。

AI API 改动：

- 验证成功响应。
- 验证 API 失败或超时时能兜底。
- 验证输出不会破坏结构化记录。

无法运行验证时，必须在交接中说明原因和风险。

## 14. 交接格式

每个 AI 完成任务后，用以下格式交接：

```text
完成内容：
- ...

修改文件：
- ...

验证：
- ...

风险 / 待确认：
- ...

下一步建议：
- ...
```

如果任务未完成，不要包装成完成。直接说明卡在哪里、已尝试什么、需要谁确认。

## 15. 禁止事项

以下行为默认禁止：

- 擅自更换项目方案。
- 擅自扩大第一阶段范围。
- 擅自删除废案、PRD 或他人改动。
- 不读取 `AGENTS.md`、`Agent.md` 和 PRD 就开工。
- 多个 AI 共用同一个任务分支并同时改核心文件。
- 完成可验证小任务后长期不提交。
- 多个 AI 共同编辑同一个 changelog。
- 让本地互动空间版依赖网络。
- 把 API Key、访问令牌或私密配置提交进仓库。
- 让 LLM 决定核心数值和结局。
- 为了炫技引入重框架、重素材或复杂架构。
- 只改代码不验证。

## 16. 当前第一阶段完成标准

第一阶段完成时，至少满足：

1. 用户能进入接案台并理解月老身份。
2. 林黛玉委托详情完整可读。
3. 候选池中孙悟空完整可选，其他候选可占位。
4. 三回合相亲均可完成地点、事件、话题选择。
5. 每回合有对话、内心 OS、数值变化和 KPI 反馈。
6. 结果页展示关系档次、月老称号、三维数值、三回合记录和最终相亲过程记录。
7. 本地版离线可跑。
8. AI 版如果接入 API，失败时仍有兜底体验。
