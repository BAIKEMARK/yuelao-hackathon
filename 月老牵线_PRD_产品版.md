# 《姻缘簿：月老牵线局》PRD（产品阐述版）

## 1. 项目概述

《姻缘簿：月老牵线局》是一款以四大名著角色为基础的轻量互动作品。用户扮演月老，接收名著人物的情感委托，在候选对象池中选择撮合对象，并通过安排相遇地点、突发事件和聊天话题，推动两位角色建立不同层次的关系。

第一阶段以“林黛玉 × 孙悟空”为完整样板线。林黛玉的委托不是“求被拯救”，而是希望遇见一个“不怕她说真话，也听得懂她话中意思的人”。孙悟空的张力在于他一开始听不懂弦外之音，但愿意认真学；他表达轻，承诺重。两人的核心爽点是反差喜剧与治愈成长并存：从鸡同鸭讲，到逐渐接住彼此。

项目同时规划两个版本：

| 版本 | 定位 | 核心能力 |
| --- | --- | --- |
| 本地互动空间版 | 黑客松提交与稳定演示 | 本地预写分支、规则数值、模板化结果记录 |
| AI API 完整版 | 展示 AI 进入核心玩法 | LLM 实时生成角色对话、内心 OS、姻缘簿相亲记录 |

两个版本共用同一套角色、选项、数值和记录协议，保证体验方向一致。

## 2. 用户对象

### 2.1 主用户

**泛娱乐轻度玩家**

- 打开后需要在数秒内理解玩法。
- 喜欢反差 CP、名著错位、轻喜剧结果。
- 有截图分享“我把林黛玉和孙悟空撮合成了什么关系”的动机。

**黑客松评委**

- 需要在 60-90 秒内体验完整闭环。
- 关注创意是否可玩、AI 是否进入核心互动、Demo 是否稳定。
- 需要看出作品具备扩展为多角色、多委托、多时间线的潜力。

### 2.2 副用户

**四大名著 / 国风梗爱好者**

- 对角色再创作、原著梗和人物反差有兴趣。
- 愿意比较不同候选对象与求助者的关系可能性。

**剧情向互动小说玩家**

- 关注分支、结局、角色内心和过程记录。
- 愿意重复体验，收集不同关系档次和相亲名场面。

## 3. 用户需求与体验动机

用户不是单纯看一段 CP 文，而是在做一次“可操作的牵线实验”：

1. 我能选择撮合谁，而不是只能接受系统固定配对。
2. 我能安排他们怎么见面、遇到什么事、聊什么话题。
3. 每一步选择后，我能看到两位角色的即时反应和内心 OS。
4. 我能看到心动、信任、默契与月老 KPI 的变化。
5. 最后我能得到一份属于这局的“姻缘簿相亲过程记录”。

核心体验目标是让用户产生：“这段关系是我亲手撮合出来的”的参与感。

## 4. 核心玩法

### 4.1 一句话玩法

用户扮演月老，从名著人物委托出发，在候选池中选择撮合对象，并通过三回合相亲安排，决定两位角色最终形成爱情、欢喜冤家、知己、礼貌收场或翻车等不同关系。

### 4.2 页面流程

```text
接案台
  -> 委托详情
  -> 候选池选择
  -> 三回合相亲
  -> 姻缘簿结果记录
```

### 4.3 三回合相亲结构

每回合用户都选择三类要素：

- 地点：他们在哪里相遇。
- 事件：他们遇到什么突发状况。
- 话题：月老引导他们聊什么。

选项密度采用递进式：

| 回合 | 地点 | 事件 | 话题 | 目的 |
| --- | --- | --- | --- | --- |
| 第 1 回合 | 2 选 1 | 2 选 1 | 2 选 1 | 建立第一印象 |
| 第 2 回合 | 2 选 1 | 2 选 1 | 2 选 1 | 制造互动与误会 |
| 第 3 回合 | 3 选 1 | 3 选 1 | 3 选 1 | 推动关系定局 |

每回合结束后，系统反馈：

- 两位角色的对话。
- 关键内心 OS。
- 心动 / 信任 / 默契变化。
- 月老 KPI 变化。
- 当前关系倾向。

## 5. 关系档次

最终关系不强行只有“恋爱成功 / 失败”，而是按自然匹配度分层。

| 档次 | 关系形态 | 说明 |
| --- | --- | --- |
| S | 爱情 / 正缘牵成 | 心动、信任、默契均达到较高水平，关系自然成立 |
| A | 欢喜冤家 / 暧昧拉扯 | 心动强，冲突也强，适合继续发展 |
| B | 知己 / 灵魂互补 | 信任和默契高，爱情未必成立，但彼此真正理解 |
| C | 礼貌收场 | 没有明显伤害，也没有形成深层连接 |
| D | 红线打结 / 强扭翻车 | 选择持续踩雷，关系明显不自然 |

产品态度是“爱情档次最高，但不强求所有关系都变成爱情”。强扭的瓜不甜，用户的任务是找到尽可能自然、动人的关系形态。

## 6. 第一阶段内容

### 6.1 完整样板线

第一阶段完整开放：

```text
求助者：林黛玉
完整候选对象：孙悟空
```

林黛玉委托核心：

> 我想见一个不怕我说真话、也听得懂我话中意思的人。

人物理解：

- 她不是求哄，也不是求被拯救。
- 她的敏锐不是“多心”，而是看得太真、听得太明白。
- 她要的是能平等接招的人：不怕她锋利，也不把她的真话当成麻烦。

孙悟空关系张力：

- 一开始听不懂弦外之音，但愿意认真学。
- 承诺很重，表达很轻。
- 他不擅长精细安慰，但不会躲、不会装懂，也不怕林黛玉的锋芒。

### 6.2 候选池

第一阶段展示 4-5 个候选对象，其中孙悟空完整可玩，其余为简版或待解锁。

建议候选池：

| 候选对象 | 状态 | 关系想象 |
| --- | --- | --- |
| 孙悟空 | 完整可玩 | 反差喜剧 + 治愈成长 |
| 贾宝玉 | 简版 / 待解锁 | 熟人局，情绪共鸣强，但可能困在旧关系 |
| 唐僧 | 简版 / 待解锁 | 稳定克制，信任高，心动慢 |
| 武松 | 简版 / 待解锁 | 安全感强，表达笨拙 |
| 猪八戒 | 简版 / 待解锁 | 热情直接，轻喜剧，可能翻车 |

## 7. 双版本体验差异

### 7.1 本地互动空间版

目标是稳定可提交、可离线体验。

- 不依赖网络通信。
- 入口为 `index.html`。
- 使用本地预写分支、角色 OS、结果记录模板。
- 通过打字机、气泡弹出、生成中状态，模拟“角色即时回应”的交互感。
- 明确说明 Demo 使用本地分支，完整版接入 LLM。

### 7.2 AI API 完整版

目标是展示 AI 进入核心玩法后的上限。

- LLM 根据角色卡、用户选择、当前数值生成对话。
- LLM 生成角色内心 OS，让用户感到角色在实时回应。
- 结尾由 LLM 整理本局选择与关键反应，生成“姻缘簿相亲过程记录”。
- 数值变化仍由规则引擎控制，避免 LLM 判断不稳定影响游戏性。

## 8. 创新优势

### 8.1 名著人物不是皮肤，而是玩法变量

角色的性格、表达方式、心结、雷区和加分点直接影响数值与剧情反馈。用户不是简单看“林黛玉和孙悟空谈恋爱”，而是在学习如何让两个表达系统完全不同的人建立关系。

### 8.2 月老视角带来强参与感

用户不是扮演其中一方，而是站在关系调度者的位置做选择。地点、事件、话题的组合让用户有“做局”和“调参”的乐趣。

### 8.3 关系结果不二元

产品不把“爱情成功”作为唯一正确答案。知己、欢喜冤家、礼貌收场和翻车都可以成为可记录、可分享的结局，降低挫败感，同时提升复玩价值。

### 8.4 AI 参与过程，而不是只生成结尾

AI API 版中，LLM 在每一回合都参与角色回应和内心 OS 生成。用户能在过程中感受到“正在和 AI 驱动的角色互动”，而不是只在结尾看到一段生成文案。

### 8.5 本地版与 AI 版可并行开发

本地版保障黑客松提交稳定性，AI 版展示创新上限。两者通过统一数据协议对齐，适合多人分工。

## 9. 八角行为分析法映射

| 驱动力 | 产品落点 |
| --- | --- |
| 史诗意义与使命感 | 用户作为月老，修复名著人物错位时空中的情感困局 |
| 发展与成就 | 每回合心动、信任、默契和 KPI 变化，最终获得牵线评级 |
| 创意授权与反馈 | 用户自由组合地点、事件、话题，并获得即时角色反应 |
| 所有权与占有欲 | 结尾生成属于本局的姻缘簿相亲过程记录和月老称号 |
| 社交影响与关联性 | 结果记录适合截图分享，引发“你能牵成什么关系”的比较 |
| 稀缺性与渴望 | 候选池与待解锁委托暗示更多名著 CP 和时间线 |
| 未知性与好奇心 | 不同组合触发不同 OS、误会、彩蛋和关系结局 |
| 损失与逃避心 | KPI 下滑、红线打结、踩雷反馈制造轻量压力，但不强惩罚 |

## 10. 分享与传播点

- “我把林黛玉和孙悟空撮合成了爱情 / 知己 / 欢喜冤家。”
- “这局月老 KPI：金牌红娘 / 误会制造机 / 红线打结师。”
- “林黛玉的内心 OS 和孙悟空的反应太反差了。”
- “同样的 CP，不同选择会走向完全不同关系。”

最终分享物建议以“姻缘簿相亲过程记录”为核心，包含：

- 求助者委托。
- 候选对象。
- 三回合关键选择。
- 角色名场面。
- 最终关系档次。
- 月老称号。

## 11. 第一阶段成功标准

第一阶段产品成功的判断标准：

1. 用户 5 秒内能理解自己是月老，要帮角色牵线。
2. 用户 60-90 秒内能完成一局林黛玉 × 孙悟空。
3. 每回合选择后都有即时角色反馈和数值反馈。
4. 结局不是单一成功/失败，而是能形成可分享的关系记录。
5. 本地版稳定可提交，AI 版能展示实时角色生成能力。
